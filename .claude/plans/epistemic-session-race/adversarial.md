# Adversarial Review: epistemic-session-race

## Verdict: REWORK

The spec's foundational empirical claim is incorrect: state.json declares "PPID = claude main process PID" CONFIRMED, but the SessionStart probe shows hook context PPID comm = `sh` (intermediate shell), not `claude`. The "Bash tool $PPID = claude main" qualifier in state.json reveals the measurement was taken from a Claude-turn context, not a hook context — these are architecturally different process hierarchies. Five findings across all three lenses (C1, C9, M5, H2, IF1 cascade) converge on this single root flaw, which propagates into a deterministic failure mode: hook writes marker keyed by short-lived sh PID; sweep correctly removes it as orphan after sh dies; SessionEnd cannot find own marker. Compounded by four post-migration consumer-breakage sites (M1-M4) where commands still treat `~/.claude/.current-session` as a file after it becomes a directory, and an unhandled `source: "clear"` (compaction) branch that orphans preflight on every `/compact`. The fix is not a discard — the design direction is sound and the empirical findings (probe artifacts, vault findings, three confirmed incidents) are real. But WU1's helper design must be expanded with a process-tree-traversal function (or switched to session_id-keyed markers) BEFORE WU2-WU5 can be correctly written on top of it. Regression target: **specify** (Stage 2). Re-validate Q2/Q3 empirically (SessionEnd stdin probe + compaction probe) before re-locking the spec.

# Stage 4 Edge Cases (vanilla)

## Verdict: NEEDS_PATCHES

The rev2 spec is structurally sound but has two spec-level gaps that will produce silent failures in production and one overclaimed guarantee that misleads implementers. The `/proc/$PID/stat` parsing fragility (E1) is the most urgent: the spec directs implementers to read field 4 from `/proc/$PID/stat` for PPid, but comm names with spaces or parentheses shift field positions, causing traversal to silently compute the wrong PPid and return an empty claude PID — which degrades all marker operations for that session without a visible error. The cross-filesystem atomic write gap (E2) is lower probability but catastrophic when it occurs: a partial marker is visible during the copy window, causing a postflight to pair against a marker that is incompletely written. Both gaps are 1–2 line fixes in WU1's implementation guidance; neither requires architectural rework. Four additional medium-priority patches (E3, E5, E6, E9) improve correctness under realistic production conditions. The remaining six cases are watch-level: spec acknowledgment is appropriate but no spec text changes required.

## Architectural Implications

**E1 (stat parsing)** implies WU1's implementation should use `/proc/$PID/status` (key-value, `PPid:` line) rather than `/proc/$PID/stat` (space-delimited, field 4). Reading PPid from `/proc/$PID/status` is: `awk '/^PPid:/{print $2}' /proc/$PID/status`. This is immune to comm-name spaces. The spec text "walk `/proc/$PID/stat` PPid chain" should be corrected to "walk `/proc/$PID/status` PPid field." Single-line fix in spec and in WU1 description. No architectural rework.

**E2 (cross-filesystem mv)** implies WU1's `epistemic_write_marker` must constrain tmpfile location. Spec currently says "atomic write via tmpfile + mv" without specifying where the tmpfile lands. Fix: `mktemp -p "$(dirname "$marker_path")"` ensures tmpfile is same-filesystem as marker. This is a one-line implementation constraint that should be added to WU1's function description to prevent the default `mktemp` behavior from introducing cross-filesystem mv.

No other edge case implies the spec's core APPROACH needs to change.
